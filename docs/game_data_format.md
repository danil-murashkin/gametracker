# Формат игровых данных

---

## Структура

В игре есть две сущности:

- **Статы** (`stats`) — состояние персонажа: здоровье, SPECIAL, радиация, флаги смерти/болезни
- **Рецепты** (`recipes`) — правила изменения статов: предметы, броня, механика мира, триггеры

### Стат (`stat`)

Стат описывает состояние персонажа или объекта. Все изменения во времени определяются наложенными рецептами.


| Поле   | Описание              |
| ------ | --------------------- |
| `id`   | идентификатор стата   |
| `name` | имя стата             |
| `desc` | краткое описание      |
| `min`  | минимальное значение  |
| `max`  | максимальное значение |
| `def`  | значение по умолчанию |
| `val`  | текущее значение      |


Пример — здоровье:

```json
{
  "id": 1,
  "name": "health",
  "min": 0,
  "max": 100,
  "def": 100,
  "val": 100
}
```



### Рецепт (`recept`)

У рецепта две части: **параметры самого рецепта** и **воздействие** (`effects`) на статы — одно или несколько.

**Порядок полей фиксирован:** сначала все параметры рецепта (`id`, `name`, `desc`, `type`, `uses`, `umax`, `sig`), затем воздействие на статы (`stat`/`val` или массив `stats`). Параметры можно **пропускать**: отсутствующее поле не задаётся. Указывать нужно только то, что реально меняет поведение.

#### Параметры рецепта


| Поле   | Описание                                                |
| ------ | ------------------------------------------------------- |
| `id`   | идентификатор рецепта                                   |
| `name` | имя рецепта                                             |
| `desc` | краткое описание                                        |
| `type` | `none`, `inst`, `temp`, `pasv`                          |
| `uses` | сколько раз рецепт ещё можно применить                  |
| `umax` | максимум возможных использований                        |
| `sig`  | подпись HMAC (8 hex = 4 байта), защита от переписывания |




#### Воздействие (`effects`)

Воздействие на один стат:


| Поле    | Описание                                                                                                       |
| ------- | -------------------------------------------------------------------------------------------------------------- |
| `stat`  | имя стата, который меняем                                                                                      |
| `val`   | коррекция: префикс `+` / `-` / `=` и **число,** `rnd(…)` **или выражение** (тот же язык, что у `if` / `while`) |
| `delay` | задержка перед началом изменений (сек); `0` — сразу                                                            |
| `coef`  | коэффициент ускорения (+) или замедления (−) хода изменений; действует и при отрицательном `val`               |
| `tick`  | интервал повторной сработки `val` (сек)                                                                        |
| `try`   | количество повторов сработки                                                                                   |
| `if`    | условие наложения (строка-выражение); ложно → эффект не добавляется                                            |
| `while` | условие действия каждый тик; ложно → эффект не действует / снимается                                           |


В рецепте задаётся **одно из двух**:

1. коротко на один стат — поля `stat` + `val` (и при необходимости `delay` / `coef` / `tick` / `try` / `if` / `while`) после параметров рецепта;
2. на несколько статов — массив `stats`: элементы той же структуры (`stat`, `val`, …).



#### Условия и выражения

На воздействии на стат можно задать `if` и/или `while`. Поля опциональны.


| Поле    | Когда проверяется           | Поведение                                   |
| ------- | --------------------------- | ------------------------------------------- |
| `if`    | в момент наложения          | ложно → эффект **не добавляется**           |
| `while` | каждый тик, пока эффект жив | ложно → эффект **не действует** / снимается |


Выражения — строки. Логика: `and`, `or`, `not` (не `&` / `|`). Сравнения и составные части — в скобках.

**Функции над статами**


| Вызов         | Смысл                                        |
| ------------- | -------------------------------------------- |
| `exs(name)`   | стат есть на персонаже (в т.ч. значение `0`) |
| `act(name)`   | стат есть и значение `!= 0`                  |
| `inact(name)` | стата нет **или** значение `== 0`            |


**Математика:** `+` `-` `*` `/`, сравнения `>` `<` `=` `!=`, `rnd(n)` — целое `0 … n−1`, `rnd(a,b)` — целое от `a` до `b` включительно (например `rnd(3,10)`).  
В выражениях допустимы имена статов (`luck`, `mod_luck`, …), например `luck + mod_luck < rnd(12)`.

В `val` после префикса операции (`+` / `-` / `=`) пишется число, `rnd(…)` **или** выражение: `"+40"`, `"=1"`, `"+rnd(10)"`, `"+rnd(3,10)"`, `"+(max_health/2)"`, `"=(luck + mod_luck)"`. Результат вычисляется в момент сработки.

**Динамические статы:** запись в `stats` может **создать** стат, которого не было в базовой модели (например `hangover_nuka`).

**Перезапуск:** повторное наложение на тот же `stat` **заменяет** предыдущий инстанс (таймеры `delay` / `tick` / `try` с нуля), а не плодит копии.

**Расшифровка CBOR**

```text
{
  "dict": ["health", "rad_now", "water", "hangover_nuka"],
  "data": {
    0: "nuka_cola",
    1: "Зарядись энергией атома!",
    2: 13,
    3: 2,
    4: [
      { 5: 128, 6: "+40" },
      { 5: 129, 6: "+100" },
      { 5: 130, 6: "+18000" },
      { 5: 131, 6: "=1",
        7: "(exs(hangover_nuka)) or (inact(party_king) and (luck + mod_luck < rnd(12)))" },
      { 5: "mod_strength", 6: "-2", 9: 600, 10: 1, 11: 7200,
        8: "act(hangover_nuka)" }
    ],
    12: "a3f19c4e"
  }
}
```

Ключи и служебные токены — **фиксированный** словарь (ID 0–127, индекс в массиве, прошит на всех устройствах). Имена статов сценария — **приложенный** `dict` (ID ≥ 128 = 128 + индекс в массиве).

**Фиксированный словарь:**

```text
"dict": ["name", "desc", "type", "uses", "stats", "stat", "val", "if", "while", "delay", "tick", "try", "sig", "inst", "exs", "inact", "act", "and", "or"]
```

**Приложенный словарь** `dict` (этот рецепт):

```text
"dict": ["health", "rad_now", "water", "hangover_nuka"]
```

Имена вне словарей (`nuka_cola`, `mod_strength`, `party_king`, `luck`, …) и строки `val` / `if` / `while` остаются UTF-8 text. Значение `type` кодируется как ID `13` (`inst`). Поля `stat` для статов из `dict` — как ID 128–131.

## Протокол

Протокол поддерживает гибкую передачу структурированных данных (статов, рецептов) в разных форматах. Все бинарные блоки (CBOR) при передаче через текстовые каналы обязательно кодируются в Base64.

На NFC-метке (≤ 512 байт) **чистый CBOR** или **чистый JSON** могут использоваться напрямую. Если формат вложен в другой контейнер — действуют правила ниже.

Структура полей стата и рецепта — в начале документа ([«Структура стата (](#структура-стата-stat-и-рецепта-recept)`stat`[) и рецепта (](#структура-стата-stat-и-рецепта-recept)`recept`[)»](#структура-стата-stat-и-рецепта-recept)).

### 1. Основные форматы обмена

1. **JSON внутри CBOR** — основной контейнер CBOR, содержит `{ "dict": [...], "data": "<строка JSON>" }`. Вложенный JSON передаётся как обычная `text string`, **без** Base64.
2. **CBOR внутри JSON** — основной контейнер JSON, содержит `{ "dict": [...], "data": "<base64-кодированный CBOR>" }`.
3. **Чистый CBOR / чистый JSON** — напрямую (NFC-метка, прошивка PDA).

Все бинарные данные (CBOR-блоки) при передаче **внутри** других структур или по текстовым каналам всегда кодируются в Base64.

```text
JSON в CBOR:  CBOR { dict: [...], data: "{\"name\":\"health\",...}" }
CBOR в JSON:  { "dict": [...], "data": "o2NldGlj..." }   ← data = Base64(CBOR)
NFC:          CBOR или компактный JSON без обёртки
```



### 2. Словарь имён параметров

Всегда первым элементом идёт словарь дополнительных имён (`dict`):

- значения с ID **0–127** — **фиксированный словарь** (базовые поля, одинаковый на всех устройствах);
- ID **≥ 128** — индекс в массиве `dict` (расширения сценария).

Внутри CBOR значения стата и рецепта идут **строго по порядку**: сначала базовые параметры (ID ≤ 127), затем дополнительные — по словарю расширений.

Такой подход даёт компактность на NFC (до 512 байт), универсальность форматов и расширяемость без потери совместимости между устройствами.
